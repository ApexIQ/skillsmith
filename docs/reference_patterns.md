# Reference Patterns

## Purpose

This document captures high-value product patterns from adjacent AI coding tools and explains how `skillsmith` should adopt them.

It is not a marketing comparison.
It is a design input for improving:
- project setup
- context management
- rules
- skills
- workflows
- multi-agent behavior
- cross-tool compatibility

## 1. Antigravity Kit

Reference:
- https://antigravity-kit.vercel.app/docs/agents
- https://antigravity-kit.vercel.app/docs/skills
- https://antigravity-kit.vercel.app/docs/workflows

### Strong patterns
- clear split between `agents`, `skills`, and `workflows`
- specialist agent roster with explicit roles
- agents declare which skills they can use
- slash-command workflows are first-class
- multi-agent coordination is explicit through an orchestrator

### What `skillsmith` should adopt
- generate a default agent roster, not just one generic setup
- define a stable orchestrator pattern
- treat workflows as user-triggered runbooks, not generic markdown blobs
- let subagents reference a limited skill set to reduce noise

## 2. Windsurf

Reference:
- https://docs.windsurf.com/windsurf/cascade/skills
- https://docs.windsurf.com/windsurf/cascade/workflows
- https://docs.windsurf.com/windsurf/cascade/agents-md

### Strong patterns
- sharp separation:
  - `Rules` = behavior
  - `Skills` = procedural expertise
  - `Workflows` = manual slash-command procedures
- progressive disclosure for skills
- strong explanation of when to use each artifact type
- directory-scoped `AGENTS.md`

### What `skillsmith` should adopt
- formalize the same taxonomy in the product model
- stop mixing rules, workflows, and skills under broad generic wording
- add progressive disclosure as a design principle
- support directory-scoped guidance, not only repo-root guidance

## 3. Zencoder

Reference:
- https://docs.zencoder.ai/rules-context/zen-rules
- https://docs.zencoder.ai/features/repo-info-agent

### Strong patterns
- `repo.md` generated by a dedicated context agent
- hierarchical rules with frontmatter and explicit scope
- rule-folder interoperability with other tools
- generated project context is treated as a reusable artifact

### What `skillsmith` should adopt
- generate a canonical machine-maintained project context file from repo inspection
- add a `repo summary` or `project context` generator command
- support interoperability by rendering rule outputs into multiple tool-native folders

## 4. Augment

Reference:
- https://docs.augmentcode.com/cli/rules
- https://docs.augmentcode.com/cli/skills

### Strong patterns
- compatible with shared formats like `CLAUDE.md`
- supports both rules and skills as separate layers
- skills follow a standard `SKILL.md` model

### What `skillsmith` should adopt
- make compatibility a first-class feature
- render shared content into multiple target ecosystems
- keep skill packaging standard and portable

## 5. Cursor

Reference:
- https://docs.cursor.com/en/context/rules
- https://cursor.com/changelog/2-4
- https://cursor.com/blog/agent-best-practices/

### Strong patterns
- project rules with clear activation modes
- skills are now dynamic and separate from always-on rules
- hooks and commands expand execution behavior without bloating baseline context

### What `skillsmith` should adopt
- support rule activation modes such as always-on, glob-scoped, manual, or agent-requested
- distinguish persistent rules from dynamic skills in both docs and generation
- consider hooks as part of future workflow automation support

## 6. Gemini / Antigravity / Android Studio

Reference:
- https://developers.google.cn/gemini-code-assist/docs/use-agentic-chat-pair-programmer
- https://developer.android.com/studio/gemini/agent-files

### Strong patterns
- hierarchical context loading from multiple directories
- local file imports inside `AGENTS.md`
- component-level instruction scoping

### What `skillsmith` should adopt
- support layered context files
- generate import-friendly instruction files
- allow directory-local rules when working in monorepos or large apps

## 7. Codex

Reference:
- https://openai.com/index/introducing-codex/

### Strong patterns
- `AGENTS.md` remains the primary repo-level instruction surface
- simple, portable, repo-native instruction model

### What `skillsmith` should adopt
- treat `AGENTS.md` as the universal portability layer
- keep tool-specific outputs derived from the same source, not authored independently

## Reference-Driven Product Changes

## A. Formalize the Artifact Model

`skillsmith` should standardize these artifact types:

- `profile`
  - source of truth for the project
- `rules`
  - behavior and constraints
- `skills`
  - reusable procedural knowledge
- `workflows`
  - reusable runbooks or slash-command procedures
- `agents`
  - specialist workers with explicit roles
- `context`
  - generated repo/project snapshot

## B. Add a Generated Project Context Artifact

Inspired by Zencoder `repo.md`.

Recommended output:
- `.agent/context/project-context.md`

Contents:
- detected stack
- package manager
- build/test commands
- repo structure
- deployment/runtime facts
- key architectural notes

This should be machine-generated and refreshable.

## C. Add Role-Aware Agent Packs

Inspired by Antigravity Kit.

Default pack:
- orchestrator
- researcher
- implementer
- reviewer

Future packs:
- frontend-specialist
- backend-specialist
- test-engineer
- debugger
- docs-writer

## D. Add Workflow Packs With Explicit Invocation Model

Inspired by Windsurf and Antigravity Kit.

Workflow packs should be:
- markdown-based
- manually invokable
- generated into tool-native command locations where applicable
- mirrored into `.agent/workflows/` as the internal source layer

## E. Add Scoped Rule Rendering

Inspired by Cursor, Windsurf, Zencoder, and Gemini.

Rules should support:
- always-on
- glob-based
- manual
- agent-requested

## F. Add Interoperable Outputs

Inspired by Augment and Zencoder.

`skillsmith` should render the same logical rule set into:
- `AGENTS.md`
- `CLAUDE.md`
- `.cursor/rules/`
- `.windsurf/rules/`
- `.zencoder/rules/`
- `GEMINI.md`

## G. Add Documentation Maintenance Workflows

Inspired by Antigravity ecosystem docs workflow patterns.

Recommended built-ins:
- `docs-audit`
- `docs-update`
- `claude-sync`
- `project-context-refresh`

## H. Add Recommendation Explainability

Inspired by curated packs and trust-scored registries.

Recommended behaviors:
- explain why a skill was ranked
- expose which curated pack contributed a recommendation
- show trust, freshness, and compatibility signals before install
- persist rationale into the lockfile for later audit
- offer a preview/report command before any install flow

## Highest-Value Next Implementations

1. Add generated `.agent/context/project-context.md`
2. Implement `init --guided` with artifact selection by project profile
3. Add workflow rendering into Claude command files and internal workflow sources
4. Add rule activation metadata
5. Add `doctor` checks for missing native path outputs
6. Add a recommendation preview/report command that explains curated-pack choices before install
