# Project Evaluation (Brutal Council)

Date: 2026-03-24
Mode: 9-role council, pruning-first, forced alignment

## 0. Feature Audit (Before Strategy)

### Keep
- Feature: Readiness wedge (`init`, `doctor`, `compose`, `report --artifact-dir`, `ready`, `audit`)
- Why it survives: This is the only end-to-end flow with measurable output, CI portability, and a plausible paid path.
- Feature: Deterministic machine-output contracts (`--json` surfaces + contract tests)
- Why it survives: Automation and CI gating break without stable machine-readable outputs.
- Feature: Local-first default operation
- Why it survives: Fast install-to-value and low adoption friction.

### Improve
- Feature: Command surface discoverability
- Problem: 30 top-level commands create decision paralysis and support burden.
- Fix: Introduce core/advanced/admin grouping in docs and command help; drive all first-run users through a single readiness path.
- Feature: `ready` orchestration internals
- Problem: Strong UX, but internal boundaries still CLI-heavy and brittle.
- Fix: Move readiness flow to stable core use-case functions; keep CLI as adapter.
- Feature: Evaluation/scorecards
- Problem: Looks like metrics theater unless tied directly to merge risk and policy enforcement.
- Fix: Bind eval/score outputs to readiness gates and repo-level pass/fail decisions.

### Kill
- Feature: Broad "agent platform" positioning in onboarding
- Why it's dead weight: No one pays for generic tooling language.
- Feature: New non-readiness command families in next 30 days
- Why it's dead weight: This is a distraction from PMF proof.
- Feature: Duplicative alias surface (`context` and `context-index` both first-class)
- Why it's dead weight: Redundant UX surface with zero incremental value.

### Simplify
- Feature: Profile state lifecycle (`profile`, `sync`, `align`)
- Problem: Three commands for one user job.
- Simpler version: Keep internals but present one guided path and explicit advanced subcommands.
- Feature: Skill lifecycle surface (`list`, `discover`, `recommend`, `roles`, `add`, `update`, `rebuild`, `lint`)
- Problem: Overlapping interfaces and maintainer-grade commands in user-facing path.
- Simpler version: Keep capabilities, demote low-value commands into admin mode.
- Feature: Governance and services (`registry`, `registry-service`, `trust-service`, `safety`)
- Problem: Enterprise complexity appears too early in beginner flow.
- Simpler version: Hide behind advanced docs and policy packs; keep wedge flow clean.

## 1. What This Project Actually Is
`skillsmith` is a Python CLI for making repositories AI-agent ready with policy checks, deterministic artifacts, and optional trust/governance layers. Right now it is a technically strong utility with early wedge potential, not yet a focused venture product.

## 2. Current State
- Code maturity: high test breadth and many production-like primitives.
- Product maturity: medium; wedge exists, but value story is diluted by command sprawl.
- Architecture maturity: medium-low for scale; domain boundaries and service durability are not enterprise-grade yet.
- GTM maturity: low; distribution loop is present but under-instrumented and under-marketed.

## 3. Core Problems (Brutal Truth)
- This is still a feature-rich toolbox pretending to be a focused product.
- Command breadth is hurting activation and comprehension.
- Current moat is weak because most primitives are reproducible.
- Service architecture is local-process oriented, not hard multi-tenant infrastructure.
- Some exposed features (for example external signer provider path) are not production-complete.

## 4. Biggest Opportunities
- Own one painful outcome: "AI-ready repo with CI evidence in under 10 minutes."
- Turn readiness artifacts into mandatory PR/branch workflows.
- Build a proprietary readiness/outcome dataset (trend, drift, policy impact) that improves over time.
- Convert from command abundance to outcome packaging.

## 5. Product Strategy
- Primary wedge: `install -> init --guided -> ready -> report --artifact-dir`.
- Product principle: fewer visible features, stronger enforced outcome.
- Keep advanced trust/registry features available, but remove them from first-run narrative.
- Ship one-click CI templates so readiness evidence is generated by default in PRs.

## 6. Technical Strategy
- Boundary reset: create explicit core use-case layer; keep CLI thin.
- Contract hardening: preserve additive-only artifact schemas and deterministic JSON behavior.
- Service hardening path:
- Replace blob-style state persistence with normalized schema + migrations.
- Remove insecure defaults outside local-dev mode.
- Implement real asymmetric OIDC/JWKS path before enterprise claims.
- Verification upgrades:
- Add CI lanes for smoke + contracts + service concurrency tests.
- Reduce monolithic test files into clearer suites.

## 7. Business Model
- OSS free: single-repo readiness baseline + local artifacts.
- Team paid: policy packs, CI enforcement templates, trend reporting, audit exports.
- Enterprise paid: SSO/SCIM, centralized policy/trust, compliance mappings, support/SLA.
- Revenue rule: charge for enforcement, governance, and cross-repo visibility, not for basic CLI commands.

## 8. Growth & Marketing
- Core loop: readiness artifact appears in PR -> reviewer sees value -> flow is copied to other repos.
- ICP: platform/DevEx leaders in teams using multiple coding assistants.
- Message to keep: "Stop AI PR chaos with deterministic readiness gates."
- Message to kill: "Generic agent platform."
- 90-day growth targets:
- Median time to first green `ready` < 10 minutes.
- `init -> ready` completion > 60% in cohort.
- At least 1.5 repo expansion per initial adopted repo.

## 9. Competitive Landscape
- IDE-native incumbents (Copilot/Cursor/Continue/OpenHands-style ecosystems) dominate discovery and habit loops.
- `skillsmith` loses if it tries to be another broad assistant framework.
- `skillsmith` can win a lane if it becomes the cross-tool readiness governance layer with audit-grade artifacts.

## 10. Risks
- Product stays useful but never budget-critical.
- Team keeps adding commands and misses PMF window.
- Determinism and trust claims overreach actual runtime guarantees.
- CI adoption stalls because gate noise is too high.
- Enterprise roadmap starts before wedge metrics are real.

## 11. Action Plan
1. Freeze non-wedge feature additions for 30 days.
2. Promote one default path everywhere: `init --guided`, `ready`, `report --artifact-dir`.
3. Add `start` style guided command to remove first-run command choice friction.
4. Add `ready ci --emit github` to auto-generate and validate gate wiring.
5. Implement `ready --fix` for high-frequency blockers.
6. Move advanced commands to explicit advanced/admin documentation tiers.
7. Add CI test matrix beyond packaging/readiness workflow.
8. Launch 10-15 design-partner pilots focused only on readiness KPIs.
9. Package paid pilot around enforcement + trend reporting + audit exports.

## 12. Feature Strategy

### Tier 1 - Must Build
- Feature: `skillsmith start` (single-command guided readiness)
- Why: This is the only thing that matters for activation.
- Impact: Major conversion lift from install to first success.
- Complexity: Medium
- Feature: `ready --fix`
- Why: Diagnostics without remediation are low-retention.
- Impact: Higher pass rate and repeat usage.
- Complexity: Medium

- Why: Team adoption requires enforceable workflow, not docs.
- Impact: Org-level spread through PR and branch policy.
- Complexity: Low/Medium

### Tier 2 - Strategic
- Feature: Readiness trend timeline from historical `scorecard.json`
- Why: Turns utility output into management signal.
- When: After Tier 1 adoption metrics are stable.
- Feature: Policy packs (`strict`, `balanced`, `fast`)
- Why: Productizes governance without command sprawl.
- When: After first paid pilots.

### Tier 3 - Drop
- Feature: New broad top-level command families
- Why: Distraction with no PMF proof.
- Feature: Additional remote provider expansion now
- Why: More surface, little wedge impact.
- Feature: Deeper hosted infra narrative in onboarding
- Why: Premature complexity and message dilution.

## 13. Feature Innovation Insights

- Borrowed ideas:
- PR artifact loops, CI summaries, enforceable branch/ruleset gates.
- Adaptations:
- Convert readiness outputs into default PR evidence (`readiness_pr.md`, `scorecard.json`, `report.json`).
- Unique edge:
- Cross-tool readiness governance with deterministic evidence, not another coding assistant UI.

## 14. Final Verdict

- REBUILD
- Reasoning:
- The product has real technical substance, but it is unfocused for venture scale.
- This feature is useless unless it drives mandatory readiness behavior in team workflows.
- No one will pay for command breadth; they pay for reduced merge risk, governance proof, and enforceable policy.
- The only viable $100M path is ruthless wedge focus, outcome instrumentation, and org-level workflow lock-in.
